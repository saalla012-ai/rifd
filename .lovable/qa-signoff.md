## قاعدة الاعتماد

لا تُغلق QA اليدوية إلا إذا تحققت الشروط التالية:

1. كل السيناريوهات الـ14 في `.lovable/qa-runbook.md` اجتازت الاختبار.
2. كل سيناريو اختُبر على 3 مقاسات: Mobile 390px، Tablet 768px، Desktop 1280px.
3. كل مقاس اختُبر على Chrome وSafari.
4. لا توجد أخطاء console حرجة.
5. أي فشل موثّق في سجل الأخطاء مع قرار واضح: مصلح / مقبول / مانع.
6. لا يتم اعتماد 100% قبل تحصين الفيديو، الحملات، النقاط، ولوحة الإدارة.

## تدقيق S1-S4 ومعالجة فجوة حرجة — 25 أبريل 2026

| السيناريو | نتيجة التدقيق | القرار |
|---|---|---|
| S1 — تسجيل جديد + onboarding | ✅ كودياً سليم | التسجيل يحوّل إلى onboarding، وonboarding يحفظ `profile.onboarded=true` ويولّد أول محتوى؛ يحتاج اختبار بريد فعلي |
| S2 — توليد نص + usage + library | ✅ كودياً سليم | `generateText` يستهلك الحصة اليومية، يسجل `record_generation`، ويظهر في المكتبة عبر `generations` |
| S3 — توليد صورة + download + library | ✅ كودياً سليم | الصورة تُحفظ في التخزين، تولّد Signed URL، وتُسجل في `generations` مع زر تنزيل في الواجهة |
| S4 — اشتراك + إيصال + تفعيل | ⚠️ وُجدت فجوة ثم أُصلحت | رفع الإيصال كان يرفع الملف ثم يفشل تحديث الطلب بسبب RLS؛ نُقل تحديث `receipt_path` إلى Server Function تتحقق من ملكية الطلب والمسار |
| TypeScript + Production build بعد الإصلاح | ✅ ناجح | الإصلاح لا يكسر البناء؛ بقي تحذير chunk size فقط |

**حكم هذه الدفعة:** تم تحسين مسار الإيراد الأكثر حساسية قبل Soft Launch. لم يتم إغلاق S4 بالكامل بعد لأن وصول Telegram، تفعيل الأدمن، وبريد `subscription-activated` يحتاج جلسة أدمن/بيئة فعلية، لكن تم إزالة عائق تقني مؤكد كان سيمنع اكتمال رفع الإيصال للمستخدم.

**نسبة التقدم بعد الإصلاح:** 99.35% جاهزية بناء وتشغيل، وقرابة 95-96% جاهزية اعتماد قبل Soft Launch.
