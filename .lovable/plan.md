سأنفذ حزمة تسليم واحدة للمراجعة الاحترافية، مع مراعاة أني الآن في وضع التخطيط فقط ولا أستطيع إنشاء ZIP أو ملفات فعلية إلا بعد موافقتك والانتقال لوضع البناء.

## ما سيتم تسليمه

1. **ملف ZIP كامل للمشروع**
   - إنشاء أرشيف واحد داخل `/mnt/documents/` باسم واضح مثل `rifd-full-project-export-2026-04-27.zip`.
   - يشمل كود التطبيق، الصفحات، المكونات، الخادم، الإعدادات، public assets، remotion، الاختبارات، ملفات التوثيق، ومجلد migrations.
   - سيتم استبعاد العناصر التي لا تصلح للتصدير أو لا يجب مشاركتها كأسرار مباشرة: `.git`, `node_modules`, `dist`, `.env`, كاشات البناء. سأضيف بدلاً منها ملف manifest يوضح أنها مستبعدة ولماذا.
   - لن أضع كلمات مرور أو مفاتيح سرية داخل الأرشيف. سأوثق أسماء الأسرار الموجودة فقط بدون قيم.

2. **تقرير Prompt History / سجل العمل**
   - إنشاء ملف Markdown داخل `/mnt/documents/` مثل `rifd-prompt-history-and-changelog.md`.
   - بما أن الوصول المباشر الكامل لكل محادثات المشروع من أول يوم غير متاح لي من أدوات هذا الوضع الحالي، سأبني التقرير من:
     - ملخصات المحادثة المتاحة في السياق الحالي.
     - ملفات التوثيق داخل `.lovable/`.
     - سجل التغييرات الموثق في التقارير وخطة V9.
     - قائمة migrations وتواريخها.
   - سأضع قسم واضح بعنوان “حدود الدقة” يوضح أن السجل الكامل الحرفي لكل Prompt قد يتطلب تصدير سجل المحادثة من واجهة Lovable/History إن لم يكن متاحاً آلياً.

3. **Schema كامل لقاعدة البيانات**
   - إنشاء ملفين:
     - `rifd-database-schema.md`: شرح عربي مبسط لكل جدول وكل عمود والغرض منه.
     - `rifd-database-schema.sql`: تجميع SQL من migrations المتاحة قدر الإمكان كمرجع تقني.
   - سأغطي الجداول الـ31 الحالية، ومنها: `profiles`, `user_roles`, `user_credits`, `credit_ledger`, `video_jobs`, `campaign_packs`, `subscription_requests`, `generations`, `plan_entitlements`, `usage_logs`, `email_send_log`, وغيرها.
   - سأوثق RLS، الدوال المهمة، التخزين، والأسرار المتاحة بالاسم فقط.

4. **تقرير الصفحات والمكونات والروابط**
   - إنشاء ملف `rifd-routes-components-inventory.md`.
   - يتضمن:
     - 79 Route file حالياً تحت `src/routes`.
     - 77 Component file حالياً تحت `src/components`.
     - 24 Server module تحت `src/server`.
   - تصنيف الروابط إلى:
     - صفحات عامة وتسويقية.
     - صفحات dashboard للمستخدم.
     - صفحات admin.
     - API/public hooks.
     - routes legacy/redirect إن وجدت مثل video provider aliases.
   - الحالة ستكون مبنية على آخر تحقق موثق: build/typecheck ناجحين سابقاً، لكن QA اليدوي الكامل 0/84 غير مغلق. لذلك سأستخدم حالات دقيقة: “موثق كودياً”، “يحتاج اختبار يدوي”، “محمي بالأدمن”، “API/Hook”، وليس ادعاء أن كل رابط مجرّب عملياً.

5. **قائمة الأخطاء والمشكلات والتحذيرات المعروفة**
   - إنشاء ملف `rifd-known-issues-and-risks.md`.
   - سيشمل على الأقل:
     - البريد/الدومين `notify.rifd.site` تم تجاوزه مؤقتاً ولم يغلق كتفعيل نهائي.
     - QA V9 اليدوي لا يزال مفتوحاً: 0/84 تمريرة موثقة رسمياً.
     - Lighthouse Mobile وإعادة القياس بعد النشر لا تزال مطلوبة.
     - Soft Launch وBeta لم يغلقا بعد كاعتماد إطلاق عام.
     - تحذير حجم chunks في build موثق كملاحظة أداء.
     - بعض تحذيرات دوال SECURITY DEFINER/صلاحيات التنفيذ تم تحسينها سابقاً لكن يلزم فحص أمني نهائي قبل GA.
     - اختبارات Safari والتجارب الحية للفيديو/الدفع/البريد/Telegram تحتاج اعتماد إنتاجي.

6. **تقرير حالة المشروع الكامل**
   - إنشاء ملف `rifd-project-status-report.md`.
   - يتضمن:
     - ما تم إنجازه.
     - ما هو قيد التنفيذ.
     - ما لم ينفذ بعد.
     - نسبة التقدم الحالية حسب الوثائق: جاهزية تقنية عالية جداً، لكن 100% الرسمي مشروط بـ QA V9 + Lighthouse + Soft Launch + Beta.
     - توصية مراجعة احترافية: ما يجب فحصه أولاً قبل أي إعلان عام.

7. **ملف فهرس موحد للتسليم**
   - إنشاء `README-REVIEW-PACKAGE.md` يشرح محتويات كل ملف وكيف يستخدمه المراجع.
   - إدراج كل التقارير داخل ZIP منفصل أو بجوار ZIP في `/mnt/documents/` حسب الأفضل للتنزيل.

## طريقة التنفيذ بعد الموافقة

1. تشغيل أوامر قراءة فقط لحصر الملفات النهائية وتوليد manifest.
2. توليد التقارير بصيغ Markdown/SQL داخل `/mnt/documents/rifd-review-package/`.
3. إنشاء ZIP للمشروع وZIP لحزمة التقارير.
4. إجراء فحص سريع للأرشيف:
   - التأكد من وجود `src/`, `supabase/`, `.lovable/`, `public/`, `remotion/`, `tests/`.
   - التأكد من عدم تضمين `.env`, `.git`, `node_modules`, `dist`.
   - التأكد من أن التقارير قابلة للقراءة وتحتوي على الجداول/الروابط/المشكلات.
5. تسليم روابط التحميل عبر `presentation-artifact`.

## حدود مهمة يجب تثبيتها قبل التنفيذ

- لا يمكنني تسليم الأسرار أو كلمات المرور داخل ZIP، وسأستبدلها بقائمة أسماء secrets فقط.
- لا يمكنني الادعاء أن كل route يعمل عملياً بدون تشغيل QA والمتصفح لكل صفحة؛ سأصنّف الحالة بدقة وفق الأدلة المتاحة.
- سجل المحادثة الحرفي منذ أول يوم قد لا يكون متاحاً بالكامل آلياً من داخل الكود؛ سأصدر أفضل سجل تدقيقي ممكن من الوثائق والسياق وسجل التغييرات، مع توضيح الفجوة إن بقيت.

بعد موافقتك سأبدأ مباشرة بإنشاء حزمة التسليم والملفات المطلوبة دفعة واحدة.
