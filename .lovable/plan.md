# الخطة المعتمدة بعد المراجعة الخبيرة: ربط Google Ultra / Flow / VEO3 دون كشف التقنية للمستخدم

## الحكم المهني المختصر

نعم، يمكن تنفيذ الربط بطريقة عملية، لكن ليس كربط مباشر آمن ومستقر مع واجهة `labs.google/fx/ar/tools/flow` نفسها؛ لأنها واجهة استخدام شخصية وليست API إنتاجية رسمية. الحل الصحيح تجارياً هو بناء طبقة وسيطة داخل رِفد تفصل بين تجربة المستخدم وبين مزودي الفيديو.

المستخدم العادي لن يرى كلمات مثل Google Ultra أو VEO3 أو Replicate. سيرى فقط خيارات طبيعية مثل:

```text
سريع
متوازن
احترافي
```

أما الأدمن فسيملك صفحة تقنية احترافية للتحكم في كل مزود، تشغيله أو إيقافه، ترتيب الأولوية، اختبار الاتصال، تحديد التكلفة بالنقاط، وتفعيل البدائل. بهذه الطريقة لا يتوقف الفيديو كله عند تعطل مزود واحد.

## قرار مهم بخصوص طلبك الأخير

سيتم إلغاء تسمية المستخدم:

```text
أفضل جودة وتجارب مميزة Ultra / VEO3
```

واستبدالها بتجربة عامة غير تقنية. أسماء المزودين ستظهر في لوحة الأدمن فقط.

---

## تحليل البدائل لـ Replicate

| المزود | الأفضلية | السعر المتوقع | التوصية |
|---|---|---:|---|
| Replicate | سهل، موجود حالياً، مناسب كبداية | غالباً أغلى قليلاً من الربط المباشر لأنه وسيط | يبقى كمزود احتياطي مستقر |
| Google Veo API الرسمي | أفضل مسار طويل الأمد إذا كان متاحاً للحساب | عادة أوضح وأقرب للتكلفة الأصلية من الوسطاء | أولوية عالية عند توفر API key رسمي |
| Google Flow Bridge | يستهلك من حسابك الشخصي Ultra/Flow | قد يكون الأرخص عليك لأنه من رصيد حسابك | مناسب كجسر يدوي/شبه تلقائي، لا ننصح بأتمتة متصفح عدوانية |
| Runway API | قوي إنتاجياً وسهل API | غالباً مرتفع حسب الثواني والجودة | بديل ممتاز للاحترافي إذا كان الهامش يسمح |
| Luma API | جودة جيدة وسرعة مرنة | يختلف حسب النموذج/الدقة | بديل جيد للمتوازن/الاحترافي |
| Kling API | جودة قوية خصوصاً الحركة | يتفاوت حسب المنصة والبلد | بديل تجريبي جيد وليس المزود الأساسي أولاً |
| Pika API/منصات وسيطة | مناسب للتجارب | متغير | احتياطي لاحق فقط |

الخلاصة: Replicate ليس دائماً الأرخص. الأفضل هندسياً هو ألا نعتمد على مزود واحد. نبني Router داخلي، ثم نفعّل المزود الأرخص/الأجود حسب النتائج الفعلية في لوحة الأدمن.

## الهدف النهائي

بناء نظام فيديو متعدد المزودين بهذا الشكل:

```text
المستخدم
  يختار: سريع / متوازن / احترافي + المقاس + المدة
        ↓
Video Provider Router
        ↓
مزود 1 يعمل؟ نعم → ينشئ المهمة
        ↓ لا
مزود 2 يعمل؟ نعم → ينشئ المهمة
        ↓ لا
مزود 3 يعمل؟ نعم → ينشئ المهمة
        ↓ لا
رد نقاط تلقائي + رسالة واضحة
```

## المرحلة 1: فصل واجهة المستخدم عن أسماء المزودين

### ما سيتم تنفيذه
1. تعديل صفحة توليد الفيديو العامة بحيث لا تعرض أسماء تقنية.
2. اعتماد خيارات مستخدم بسيطة:
   - سريع: مناسب للمحتوى اليومي والتجربة.
   - متوازن: جودة أعلى مع تكلفة وسط.
   - احترافي: أفضل جودة متاحة حسب المزود النشط.
3. إخفاء Google Ultra / Flow / VEO3 / Replicate تماماً عن المستخدم العادي.
4. عرض التكلفة بالنقاط بوضوح قبل التوليد.
5. استمرار المقاسات والمدد كما هي، مع منع أي خيار غير مدعوم حسب المزود المختار داخلياً.

### معيار القبول
- المستخدم لا يرى أي اسم مزود.
- لا تظهر عبارة Ultra/VEO3 في الصفحة العامة.
- المستخدم يفهم التكلفة والجودة فقط.

---

## المرحلة 2: إنشاء مركز إدارة مزودي الفيديو للأدمن

### الصفحة الجديدة
إضافة صفحة أدمن:

```text
/admin/video-providers
```

### وظائف الصفحة
1. عرض كل مزود فيديو في بطاقة مستقلة.
2. تفعيل/إيقاف المزود.
3. تحديد هل المزود عام أم داخلي فقط.
4. ترتيب أولوية المزودين.
5. تحديد أي مستوى جودة يخدمه المزود:
   - سريع
   - متوازن
   - احترافي
6. تحديد تكلفة النقاط لكل مدة:
   - 5 ثوانٍ
   - 8 ثوانٍ
7. زر “اختبار الاتصال”.
8. عرض آخر خطأ وآخر نجاح.
9. وضع المزود في حالة:
   - نشط
   - متوقف
   - تجربة
   - يحتاج تدخل يدوي
   - غير صحي

### المزودون الذين سنجهزهم في المركز
```text
1) Replicate الحالي
2) Google Flow Bridge لحسابك الشخصي
3) Google/Veo API الرسمي عند توفر المفتاح
4) Runway API
5) Luma API
6) Kling API
```

لن نطلب كل المفاتيح فوراً. سنبني المركز بحيث يقبلها تدريجياً.

---

## المرحلة 3: Google Flow Bridge لحسابك الشخصي

### لماذا Bridge وليس ربط متصفح مباشر؟
لأن واجهة Flow شخصية، وأتمتة المتصفح/الكوكيز قد تكسر الحساب أو تتوقف عند CAPTCHA أو تغيير الواجهة. أنت قلت إنك تتحمل مسؤولية حسابك، لكن مهنياً الأفضل أن نبدأ بجسر آمن لا يخرق التجربة ولا يضع حسابك في خطر مباشر.

### طريقة العمل الأولى: جسر يدوي مضبوط
1. المستخدم يطلب فيديو “احترافي”.
2. النظام يخصم النقاط وينشئ `video_job` بحالة تحتاج تنفيذ من الأدمن إذا اختار الراوتر Google Flow Bridge.
3. في صفحة الأدمن تظهر:
   - prompt جاهز للنسخ.
   - المقاس.
   - المدة.
   - بيانات المستخدم والرصيد.
4. الأدمن يفتح حسابه في Flow، يولد الفيديو، ثم يلصق رابط النتيجة أو يرفع/يربط الملف.
5. النظام يغير المهمة إلى مكتملة، وتظهر للمستخدم.
6. إذا فشل التنفيذ، الأدمن يضغط “رد النقاط” أو النظام يردها تلقائياً بعد مهلة.

### طريقة العمل الثانية لاحقاً: جسر شبه آلي
نضيف endpoint آمن يستقبل نتيجة من وسيط خارجي خاص بك:

```text
/api/public/video-provider-callback
```

بشرط توقيع سري. هذا يسمح مستقبلاً ببناء عامل خارجي منفصل يتعامل مع حساب Flow، دون أن نربط الكوكيز داخل تطبيق رِفد.

## المرحلة 4: تعديل بنية الخادم إلى Video Provider Router

### الوضع الحالي
المنطق الحالي مرتبط مباشرة بـ Replicate داخل `src/server/video-functions.ts`.

### التعديل المطلوب
استبداله بطبقة داخلية:

```text
VideoProviderRouter
  ├─ ReplicateProvider
  ├─ GoogleFlowBridgeProvider
  ├─ GoogleVeoApiProvider
  ├─ RunwayProvider
  ├─ LumaProvider
  └─ KlingProvider
```

### القاعدة الأساسية
كل مزود يجب أن يلتزم بنفس العقد:

```text
createJob(input) → providerJobId/status/resultUrl/manualRequired
refreshJob(job) → status/resultUrl/error
testConnection() → ok/error/latency
```

بهذا لو تعطل مزود، لا يسقط النظام كله.

## المرحلة 5: قاعدة البيانات والإعدادات

### الجداول/التعديلات المقترحة
إضافة جدول جديد:

```text
video_provider_configs
```

يحتوي على:
- provider_key
- display_name_admin
- enabled
- public_enabled
- supported_quality
- priority
- cost_5s
- cost_8s
- supports_9_16
- supports_1_1
- supports_16_9
- supports_starting_frame
- mode: api / bridge / manual
- health_status
- last_success_at
- last_error_at
- last_error_message
- metadata

وتوسيع بيانات `video_jobs` عند الحاجة لإضافة:
- provider_attempts داخل metadata.
- manual_required أو حالة مشابهة إذا كان enum يسمح، أو نستخدم status الحالي مع provider واضح وmetadata.

### RLS
- القراءة والإدارة لهذه الإعدادات للأدمن فقط.
- لا يقرأ المستخدم أسرار أو أسماء مزودين.
- وظائف الخادم فقط هي التي تختار المزود.

## المرحلة 6: منع توقف الفيديو عند تعطل مزود واحد

### آلية Failover
عند طلب فيديو:
1. يقرأ الراوتر المزودين النشطين حسب الجودة المطلوبة.
2. يبدأ بالأولوية الأعلى.
3. إذا فشل مزود قبل إنشاء مهمة فعلية، ينتقل للمزود التالي.
4. إذا أنشأ المزود مهمة خارجية، لا ننقل تلقائياً إلا وفق سياسة واضحة حتى لا يتم توليد فيديوهين وخصم تكلفة مزدوجة.
5. إذا فشل كل شيء، يتم رد النقاط وإعادة الحصة اليومية.

### السياسة الافتراضية
- فشل قبل إرسال الطلب للمزود: جرّب البديل فوراً.
- فشل بعد إنشاء job عند المزود: انتظر refresh أو timeout ثم رد النقاط.
- Google Flow Bridge: لا يمنع بقية المزودين من العمل، لكنه قد ينتج حالة “ينتظر تنفيذ الأدمن”.

## المرحلة 7: لوحة مهام الفيديو الحالية

تطوير صفحة:

```text
/admin/video-jobs
```

لتعرض:
1. المزود المستخدم.
2. هل المهمة API أم Bridge.
3. prompt للنسخ.
4. زر فتح Flow.
5. زر لصق/حفظ رابط النتيجة.
6. زر رد النقاط.
7. سجل المحاولات.
8. سبب الفشل وآخر رسالة من المزود.

## المرحلة 8: الأسرار والمفاتيح

الموجود حالياً:
```text
REPLICATE_API_TOKEN
```

سنحتاج لاحقاً، حسب ما تريد تفعيله:
```text
GOOGLE_GEMINI_API_KEY أو Google/Veo API credentials
RUNWAY_API_KEY
LUMA_API_KEY
KLING_API_KEY
VIDEO_PROVIDER_CALLBACK_SECRET
```

لن نخزن أي كوكيز لحساب Google Flow داخل الواجهة العامة. وإذا تم بناء وسيط خارجي لاحقاً، سيكون توقيعه عبر secret مستقل.

## ترتيب التنفيذ المقترح

```text
1) إخفاء أسماء المزودين من صفحة المستخدم واعتماد سريع/متوازن/احترافي.
2) إضافة جدول إعدادات مزودي الفيديو مع RLS للأدمن.
3) بناء VideoProviderRouter مع Replicate كمزود قائم.
4) إضافة Google Flow Bridge كمسار يدوي/شبه آلي.
5) بناء صفحة /admin/video-providers.
6) تطوير /admin/video-jobs لتنفيذ مهام Bridge والرد اليدوي/الإكمال اليدوي.
7) إضافة callback آمن للوسيط الخارجي مستقبلاً.
8) تجهيز أماكن Runway/Luma/Kling دون تفعيلها حتى توضع مفاتيحها.
9) اختبار الخصم، الرد، timeout، وفشل مزود مع استمرار البدائل.
```

## معيار النجاح النهائي

نعتبر المهمة ناجحة عندما يتحقق التالي:

```text
- المستخدم لا يرى Google/Flow/VEO3/Replicate.
- الأدمن يستطيع تفعيل/إيقاف وترتيب مزودي الفيديو.
- Replicate الحالي يستمر بالعمل ولا ينكسر.
- Google Flow Bridge متاح كمسار احترافي من لوحة الأدمن.
- أي فشل قبل التنفيذ ينتقل إلى بديل إن وجد.
- أي فشل بعد الخصم يرد النقاط مرة واحدة فقط.
- لا تتوقف كل الفيديوهات بسبب مزود واحد.
- صفحة الأدمن تعرض التكلفة، المزود، الحالة، وسجل الأخطاء بوضوح.
```

هذه هي الخطة التي أنصح بتنفيذها. هي أعمق وأأمن من ربط حساب Flow مباشرة داخل التطبيق، ومع ذلك تحقق هدفك: استخدام حسابك Google Ultra/Flow بطريقة وسيطة، مع إبقاء المنتج احترافياً للمستخدم وقابلاً للتوسع على أكثر من API.
