# الخطة النهائية الموثقة لاستكمال إطلاق V9 بعد الفيديو واستوديو الحملات

## 1. الحكم التنفيذي بعد المراجعة

الحالة الحالية ليست جاهزية إطلاق عام، بل جاهزية بناء متقدمة. الوثائق تذكر 98.7%، لكن الفحص الحالي كشف فجوة لا يجوز تجاهلها قبل الاعتماد:

| المحور | الحكم المهني |
|---|---|
| المنتج الأساسي | مستقر فنياً، لكن الاعتماد العام مشروط بـQA كامل |
| توليد الفيديو | منطق الخادم جيد: خصم، حد يومي، حد تزامن، refund، polling. يحتاج اختبار إنتاجي موثق |
| استوديو الحملات | موجود ومحفوظ، لكن الربط العميق بالمخرجات والمكتبة ما زال جزئياً |
| النقاط والLedger | جيد للفيديو، لكن يلزم اختبار حالات الفشل والتكرار والاسترداد عملياً |
| الإدارة وAudit log | موجودة جزئياً، وتحتاج توحيد تغطية كل الأفعال الحساسة |
| التريجرات | توجد دوال وتشير الوثائق إلى تفعيلها، لكن الفحص الحالي لقاعدة البيانات يعرض: لا توجد تريجرات حالياً. هذه فجوة حرجة قبل أي إطلاق |
| QA V9 | 0/84 موثق، وهي أكبر بوابة متبقية |
| Lighthouse/SEO | مفتوح ولم يغلق |

القرار: لا أعتمد الإطلاق العام الآن. أعتمد مسار إغلاق نهائي مضبوط يبدأ بإصلاح فجوة التريجرات فعلياً، ثم إغلاق الربط العميق للحملات، ثم QA V9، ثم Lighthouse، ثم Soft Launch.

## 2. خريطة الطريق الرسمية إلى 100%

```text
المرحلة A  تصحيح البنية الحرجة        → تفعيل/إثبات التريجرات فعلياً
المرحلة B  إغلاق الفيديو والنقاط       → اختبار إنتاجي للفشل/النجاح/الاسترداد/الحدود
المرحلة C  إكمال Campaign Studio       → ربط الحملات بالمخرجات والمكتبة والتتبع
المرحلة D  توحيد Admin + Audit          → كل إجراء حساس قابل للتدقيق
المرحلة E  QA V9 الكامل                 → 84 تمريرة موثقة
المرحلة F  Lighthouse + SEO + مراقبة    → أرقام أداء وقابلية اكتشاف
المرحلة G  Soft Launch محدود            → 10-25 مستخدماً مع مراقبة تكلفة وفشل
المرحلة H  Beta Launch                   → 50-100 مستخدم بعد إصلاح ملاحظات Soft Launch
المرحلة I  GA                            → اعتماد عام فقط بعد تحقق البوابات
```

## 3. خطة التنفيذ المعتمدة للمرحلة القادمة

### المرحلة A — إصلاح فجوة التريجرات وإثباتها

الأولوية الأولى لأنها تؤثر على التسجيل، النقاط، الاشتراكات، التوليدات، والتنبيهات.

سيتم تنفيذ التالي:
1. إنشاء migration إصلاحي idempotent يعيد تفعيل التريجرات المطلوبة فعلياً:
   - `on_auth_user_created` على نظام المستخدمين لتشغيل `handle_new_user`.
   - تريجرات `updated_at` للجداول التشغيلية المهمة.
   - `trg_enforce_generation_integrity` و`trg_enforce_generation_quota` على `generations`.
   - تريجرات الاشتراكات: مزامنة الخطة، reset credits، وتنبيه الإدارة.
   - تريجر `lock_topup_from_package` على topup purchases.
   - تريجر `daily_video_usage.updated_at`.
2. إزالة أي أسماء قديمة أو مكررة قبل الإنشاء لمنع التنفيذ المزدوج.
3. بعد التنفيذ، التحقق باستعلام فعلي من قاعدة البيانات أن التريجرات موجودة، وليس الاكتفاء بالوثائق.
4. تحديث الوثائق لتقول: “مثبتة فعلياً بعد فحص information_schema” فقط إذا ظهرت فعلاً.

معيار القبول:
- استعلام التريجرات يرجع القائمة المتوقعة.
- لا توجد أسماء مكررة لنفس الوظيفة على نفس الجدول.
- التسجيل الجديد يخلق profile وcredits وrole تلقائياً.

### المرحلة B — إغلاق الفيديو والنقاط إنتاجياً

سيتم تدقيق وتنفيذ اختبار فعلي للمسارات التالية:
1. نجاح توليد فيديو: خصم مرة واحدة، إنشاء job، ظهور النتيجة، تحديث الرصيد.
2. فشل توليد فيديو: refund مرة واحدة، status = refunded، ledger واضح.
3. تجاوز الحد اليومي: منع قبل الخصم.
4. حد المهام المتزامنة: منع المهمة الثالثة عند وجود مهمتين processing.
5. timeout: تحويل المهمة إلى refunded بعد المدة المحددة.
6. رسائل المستخدم عربية وآمنة، والتفاصيل التقنية في metadata/admin فقط.

معيار القبول:
- لا خصم مزدوج.
- لا refund مزدوج.
- لا processing أبدي دون معالجة.
- كل عملية قابلة للتفسير من `video_jobs` و`credit_ledger`.

### المرحلة C — تحويل Campaign Studio إلى أصل إنتاجي كامل

الوضع الحالي جيد كبداية، لكنه يحتاج ربطاً أعمق حتى لا تكون الحملة مجرد prompt محفوظ.

سيتم تنفيذ التالي:
1. إضافة ربط صريح بين Campaign Pack والمخرجات:
   - `campaign_pack_id` أو metadata موحدة في مخرجات النص/الصورة/الفيديو حسب أنسب بنية حالية.
2. عند توليد نص/صورة/فيديو من الحملة، يتم حفظ العلاقة بحيث تظهر في المكتبة كسياق حملة.
3. تحسين صفحة الحملة لعرض “مخرجات هذه الحملة” إن أمكن دون كسر المكتبة الحالية.
4. اختبار Mobile/Tablet/Desktop لحقول الحملة الطويلة، البطاقات، أزرار التوليد، وقائمة الحملات.
5. إزالة أي كود قديم أو مكرر إن ظهر أثناء الربط.

معيار القبول:
- المستخدم ينشئ حملة، يحفظها، يحدث الصفحة، يعود إليها، ويرى مخرجاتها مرتبطة.
- لا overflow على الهاتف.
- المكتبة تعرض المخرجات دون تسريب بين المستخدمين.

### المرحلة D — توحيد Audit log والإدارة

سيتم إغلاق تغطية سجل التدقيق على كل إجراء إداري مؤثر:
1. تفعيل/تعديل الاشتراكات.
2. تعديل حدود الخطط أو النقاط.
3. تفعيل/رفض/استرداد topups.
4. تحديث حالات رسائل التواصل.
5. أي تعديل إداري على Campaign Packs إن وُجد.
6. توحيد helper داخلي لتسجيل audit بدل تكرار منطق التسجيل في عدة ملفات.

معيار القبول:
- كل إجراء إداري حساس ينتج عنه before/after/metadata.
- كل صفحات الإدارة تمر عبر `assertAdmin` خادمياً.
- لا اعتماد على localStorage أو صلاحية من الواجهة.

### المرحلة E — QA V9 الرسمي: 84 تمريرة

بعد تثبيت المراحل السابقة، يتم تنفيذ المصفوفة كما هي:

| النطاق | العدد |
|---|---:|
| السيناريوهات | 14 |
| المقاسات | 3: Mobile 390، Tablet 768/820، Desktop 1280 |
| المتصفحات | Chrome وSafari |
| الإجمالي | 84 تمريرة |

السيناريوهات الأساسية:
- التسجيل، onboarding، dashboard.
- توليد نص وصورة وفيديو.
- المكتبة والusage.
- الاشتراك والإيصال والتفعيل.
- contact وTelegram/email/admin inbox.
- تجاوز الحصة والترقية.
- Campaign Pack والحفظ والتوليد منها.
- Admin للفيديو والحملات والledger.

معيار القبول:
- توثيق النتائج داخل `.lovable/qa-signoff.md`.
- أي فشل يصنف: مانع / مصلح / مقبول بسبب موثق.
- لا رفع إلى 100% قبل إغلاق هذه المصفوفة.

### المرحلة F — Lighthouse وSEO والمراقبة

سيتم فحص الصفحات:
- `/`
- `/pricing`
- `/about`
- `/auth`
- `/contact`
- `/dashboard`
- `/dashboard/generate-video`
- `/dashboard/campaign-studio`

معايير القبول:
- الصفحات العامة: Performance ≥80، Accessibility ≥90، Best Practices ≥90، SEO ≥95.
- صفحات dashboard: Performance ≥75، Accessibility ≥90، Best Practices ≥90.
- لا console errors حرجة.
- مراجعة metadata للصفحات العامة.

### المرحلة G — Soft Launch محدود

النطاق المقترح:
- 10 إلى 25 مستخدماً فعلياً.
- مدة 3 إلى 7 أيام.
- قياس: فشل الفيديو، زمن المعالجة، refund، استهلاك النقاط، إنشاء الحملات، التحويل للاشتراك، شكاوى المستخدمين.

معيار القبول:
- لا فشل حرج يمنع الاستخدام.
- تكلفة الفيديو ضمن المتوقع.
- لا شكاوى متكررة من مسار الدفع أو الحصة أو الحفظ.

### المرحلة H — Beta Launch

النطاق المقترح:
- 50 إلى 100 مستخدم.
- تفعيل مراقبة يومية للإدارة.
- معالجة ملاحظات Soft Launch قبل توسيع الوصول.

معيار القبول:
- استقرار لمدة أسبوع دون incident حرج.
- وضوح funnel الاشتراك.
- قدرة الإدارة على التشغيل دون تدخل تقني مستمر.

### المرحلة I — GA: الإطلاق العام

لا يتم اعتماد GA إلا إذا:
1. التريجرات مثبتة فعلياً.
2. الفيديو والنقاط اجتازا اختبارات النجاح والفشل والاسترداد.
3. Campaign Studio مرتبط بالمخرجات والمكتبة.
4. Audit log يغطي الأفعال الحساسة.
5. 84/84 تمريرة QA موثقة أو أي استثناء موثق وغير مانع.
6. Lighthouse ضمن الحدود.
7. Soft Launch وBeta بلا blockers.

## 5. ترتيب التنفيذ الفوري بعد اعتمادك

سأبدأ بالترتيب التالي:
1. إصلاح وإثبات التريجرات فعلياً في قاعدة البيانات.
2. تحديث الوثائق لتطابق الواقع لا الافتراض.
3. إغلاق ربط Campaign Pack بالمخرجات والمكتبة.
4. توحيد Audit log حيث توجد فجوات.
5. تشغيل فحوص TypeScript/build/security.
6. توثيق النتيجة ونسبة التقدم التالية، ثم عرض الخطوة التالية بعد التنفيذ.

هذه هي الخطة النهائية الموصى بها لمركز البناء. الخطوة التالية التنفيذية الأولى هي المرحلة A لأنها شرط أمان وتشغيل قبل أي QA واسع أو إطلاق محدود.
