# وثيقة اعتماد التنفيذ — خطة استكمال إطلاق SaaS بعد إضافة توليد الفيديو واستوديو الحملات

## 1. قرار الاعتماد

تم اعتماد الخطة التنفيذية التالية كمرجع رسمي لمركز البناء، بهدف الوصول بالمنصة إلى جاهزية إطلاق إنتاجية مستقرة بعد إضافة:

1. توليد الفيديو.
2. استوديو الحملات الإعلانية.
3. لوحة إدارة قوية وآمنة للمميزات الجديدة.
4. نظام نقاط مضبوط ومراقب وقابل للتدقيق.
5. مصفوفة QA موسعة قبل الإطلاق العام.

الحالة الحالية المعتمدة للتنفيذ: **99.86% جاهزية بناء وتشغيل — غير معتمدة للإطلاق العام حتى إغلاق QA V9 وSoft Launch**.

سبب إعادة ضبط الجاهزية من 97% إلى 88% عند اعتماد V9: الجاهزية السابقة كانت تخص المنتج الأساسي قبل إدخال توليد الفيديو واستوديو الحملات. بعد تنفيذ مراحل التحصين والإدارة والأمان والفحص التقني الداعم وحدود الفيديو اليومية الخادمية، ثم تنظيف تكرار التريجرات وتدعيم Audit log، ثم التأكد من تفعيل التريجرات الحرجة فعلياً وإزالة التكرارات القديمة ومعايرة مراقبة الإساءة حسب حدود الباقة، ارتفعت الجاهزية التنفيذية إلى 99.86%، مع بقاء الإطلاق العام مشروطاً بإغلاق QA V9 اليدوي.

---

## 2. الهدف النهائي

إطلاق SaaS مستقر تجارياً وتقنياً، بحيث تكون المنصة قادرة على:

- استقبال مستخدمين حقيقيين دون كسر تجربة المنتج الأساسية.
- توليد نصوص، صور، وفيديوهات ضمن نظام نقاط عادل وآمن.
- إنشاء حملات إعلانية منظمة وليست مجرد prompts منفصلة.
- تمكين الإدارة من مراقبة التكلفة، الاستخدام، الأخطاء، الاستردادات، والمحتوى.
- منع الاستنزاف المالي أو إساءة الاستخدام.
- توثيق الجاهزية عبر QA يدوي ونتائج بناء وفحوص أداء وأمان.

---

## 3. المبادئ الحاكمة للتنفيذ

| المبدأ | القرار |
|---|---|
| السلامة قبل السرعة | لا يتم إطلاق الفيديو والحملات للعامة قبل QA موسع |
| الخادم مصدر الحقيقة | أسعار النقاط والخصومات والاستردادات لا تعتمد على الواجهة |
| عدم كسر المنتج الأساسي | أي تعديل يجب ألا يضر النصوص، الصور، الاشتراكات، المكتبة، أو billing |
| الإدارة أولاً | لا تُطلق ميزة مكلفة دون أدوات مراقبة وتحكم Admin |
| التدرج في الإطلاق | Soft Launch ثم Beta ثم GA |
| التوثيق إلزامي | لا تُرفع الجاهزية إلى 100% إلا بعد توثيق الاختبارات |

# المرحلة 0 — مصدر الحقيقة والتوثيق

## الهدف

## التنفيذ

1. إنشاء/تحديث وثيقة رسمية باسم:
   - `.lovable/saas-launch-roadmap-v9.md`

2. توثيق الحالة الرسمية:
   - الجاهزية الحالية: 99.86% للبناء والتشغيل، وليست اعتماد إطلاق عام.
   - المنتج الأساسي: مستقر فنياً.
   - توليد الفيديو: يحتاج تحصين إنتاجي.
   - استوديو الحملات: يحتاج تحويله إلى نظام محفوظ ومترابط.
   - QA اليدوي: غير مغلق بعد.

3. تحديث تقرير الإصدار النهائي لاحقاً حتى لا يذكر 97% كجاهزية كلية بعد إضافة الفيديو والحملات، بل يوضح الفرق بين:
   - Core SaaS readiness.
   - New Features readiness.
   - Overall Launch readiness.

## معيار القبول

- لا توجد وثيقة تعلن 100% قبل QA.
- لا يوجد تضارب بين roadmap، final release report، runbook، وqa-signoff.

# المرحلة 1 — تحصين توليد الفيديو

## الهدف
تحويل توليد الفيديو من ميزة أولية إلى ميزة إنتاجية مستقرة وقابلة للدفع والمراقبة.

## نقاط المراجعة الحرجة

| الجانب | القرار |
|---|---|
| مدة الفيديو | توحيد القيم بين الواجهة، الخادم، وقاعدة البيانات |
| الحالة | إضافة polling واضح بدلاً من ترك المستخدم يحدّث يدوياً |
| الأخطاء | عرض رسائل عربية مفهومة بدلاً من أخطاء تقنية خام |
| الاسترداد | رد النقاط تلقائياً عند فشل التوليد |
| التكلفة | تسجيل التكلفة المقدّرة ومقارنتها بالاستهلاك |
| المكتبة | حفظ الفيديو الناتج وإظهاره للمستخدم |

## التنفيذ

1. مراجعة `src/server/video-functions.ts`.
2. مراجعة `src/routes/dashboard.generate-video.tsx`.
3. توحيد `durationSeconds` مع قيود قاعدة البيانات.
4. إضافة آلية تحديث حالة تلقائية للمهام قيد المعالجة.
5. تحسين رسائل الحالات:
   - قيد الإنشاء.
   - اكتمل.
   - فشل وتم رد النقاط.
   - مزوّد الفيديو غير متاح مؤقتاً.
6. منع الخصم المكرر عند إعادة المحاولة أو تحديث الصفحة.
7. التأكد من أن `video_jobs` يسجل:
   - المستخدم.
   - prompt.
   - الجودة.
   - المدة.
   - الحالة.
   - النقاط المخصومة.
   - رابط النتيجة.
   - رسالة الخطأ إن وجدت.

## معيار القبول

- لا يوجد اختلاف بين UI والخادم وقاعدة البيانات.
- عند فشل التوليد، تعود النقاط ولا يبقى job بوضع مضلل.
- المستخدم يفهم حالة الفيديو دون معرفة تقنية.
- الإدارة تستطيع رؤية حالة كل job لاحقاً.

# المرحلة 2 — نظام النقاط والتكلفة

## الهدف
ضمان أن نظام النقاط آمن، عادل، قابل للتدقيق، ولا يسبب خسارة تشغيلية.

## القرار التنفيذي

نظام النقاط يجب أن يكون محسوباً من الخادم فقط، وليس من الواجهة. الواجهة تعرض الأسعار، لكنها لا تقررها.

## التنفيذ

1. مراجعة `src/server/credits.ts` كمصدر مركزي للتكلفة.
2. تثبيت تكلفة الفيديو حسب الجودة والمدة.
3. منع أي توليد إذا لم يكن لدى المستخدم رصيد كافٍ.
4. تسجيل كل عملية في ledger واضح:
   - استهلاك.
   - استرداد.
   - سبب الاسترداد.
   - نوع المورد: text / image / video / campaign.
5. إضافة قواعد منع الإساءة:
    - حد يومي لتوليد الفيديو حسب الخطة — تم تفعيله خادمياً.
    - حد مهام متزامنة للمستخدم — تم تفعيله خادمياً.
   - منع spam requests.
6. مراجعة تجربة تجاوز الرصيد وربطها بالترقية.

## معيار القبول

- لا يمكن توليد فيديو دون رصيد كافٍ.
- لا يمكن خصم النقاط مرتين لنفس المهمة.
- أي فشل ينتج عنه refund واضح وقابل للتدقيق.
- تكلفة الفيديو لا تعتمد على مدخلات العميل وحده.

---

# المرحلة 3 — استوديو الحملات الإعلانية

## الهدف
تحويل استوديو الحملات من مولّد prompt إلى نظام Campaign Pack حقيقي.

## Campaign Pack يجب أن يحتوي على

1. اسم الحملة.
2. هدف الحملة.
3. الجمهور المستهدف.
4. العرض أو المنتج.
5. النبرة التسويقية.
6. Hook رئيسي.
7. نصوص قصيرة للإعلانات.
8. أفكار صور.
9. أفكار فيديو.
10. CTA.
11. خطة نشر 3 إلى 7 أيام.
12. ربط بالمخرجات المحفوظة في المكتبة.

## التنفيذ

1. مراجعة `src/routes/dashboard.campaign-studio.tsx`.
2. إنشاء نموذج Campaign Pack واضح.
3. حفظ الحملات للمستخدم بدلاً من فقدانها بعد الجلسة.
4. ربط الحملة بأدوات:
   - توليد النص.
   - توليد الصورة.
   - توليد الفيديو.
5. إضافة حالات:
   - draft.
   - generated.
   - archived.
6. تحسين الواجهة على الهاتف والتابلت والديسكتوب.

## معيار القبول

- المستخدم يستطيع إنشاء حملة وحفظها والعودة إليها.
- المخرجات ليست عشوائية، بل مرتبطة بهدف الحملة.
- يمكن استخدام الحملة كنقطة انطلاق للنص والصورة والفيديو.

---

# المرحلة 4 — لوحة إدارة المميزات الجديدة

## الهدف
بناء إدارة قوية وآمنة لتوليد الفيديو والحملات.

## صفحات الإدارة المطلوبة

| الصفحة | الوظيفة |
|---|---|
| Admin Video Jobs | عرض كل مهام الفيديو وحالاتها وتكلفتها وأخطاءها |
| Admin Campaigns | عرض الحملات المنشأة ومراقبة الاستخدام |
| Admin Credit Ledger | تتبع الاستهلاك والاسترداد |
| Admin Abuse Monitor | كشف الاستخدام المفرط أو المشبوه |
| Admin Cost Overview | تقدير التكلفة التشغيلية مقابل الإيراد |

## متطلبات الأمان

1. كل صفحات الإدارة محمية بـ AdminGuard.
2. الصلاحيات لا تعتمد على localStorage أو بيانات عميل قابلة للتلاعب.
3. التحقق من دور الأدمن يجب أن يتم من الخادم.
4. تسجيل الإجراءات الإدارية في audit log.
5. منع ظهور بيانات حساسة لغير المصرح لهم.

## معيار القبول

- الأدمن يرى ما يكفي لإدارة التشغيل دون دخول يدوي لقاعدة البيانات.
- غير الأدمن لا يستطيع الوصول للبيانات حتى لو عرف الرابط.
- كل إجراء مهم قابل للتدقيق.

# المرحلة 5 — مراجعة الأمان والصلاحيات

## الهدف
إغلاق فجوات الخصوصية، الملكية، والتلاعب.

## الفحوص المطلوبة

1. مراجعة سياسات الوصول للجداول الجديدة أو المعدلة.
2. التأكد أن المستخدم لا يرى إلا بياناته.
3. التأكد أن الأدمن يرى ما يلزمه عبر صلاحيات صحيحة.
4. حماية public endpoints.
5. منع إدخال روابط أو prompts تسبب مشاكل عرض أو حقن.
6. مراجعة التخزين والروابط الناتجة للفيديو.

## معيار القبول

- لا يوجد تسريب بين المستخدمين.
- لا يوجد اعتماد على صلاحيات من الواجهة فقط.
- لا توجد عمليات مكلفة دون تحقق.

5. توليد فيديو ناجح.

12. تجاوز النقاط.

## المتصفحات

## معيار القبول

## 8. القرار النهائي

الخطوة التنفيذية التالية عند الانتقال إلى وضع البناء:

1. حفظ هذه الوثيقة داخل المشروع كملف رسمي.
2. إبقاء التقارير متوافقة مع نسبة 98.7% الحالية بعد تحصين الفيديو والحملات والإدارة والأمان والتريجرات وحدود الفيديو اليومية، وتنفيذ QA بصري سريع على الهاتف والتابلت والديسكتوب.
3. تنفيذ المرحلة 6: QA V9 الموسع دون إعادة تخطيط.
4. بعد كل تمريرة QA، يتم تحديث سجل QA ونسبة التقدم.
